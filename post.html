Why most WordPress themes fail their editors

March 9, 2025 12 min read
Featured Image · 2.4:1

The block editor changed everything — except the way most themes are built. After three years building native block themes, here is what I have learned about making WordPress actually work for the people who use it every day.

Most WordPress themes are built for developers. The block editor was supposed to change that — and in many ways it has. But the themes that ship with it, and the custom themes built on top of it, still carry assumptions that belong to a different era of WordPress development.

The core assumption is this: that the person building the site and the person editing the site are the same person, or at least that they share the same mental model. They almost never do.

The editor's perspective

Editors do not think in terms of blocks. They think in terms of content: here is a headline, here is a paragraph, here is a pull quote. The block editor's promise was to make those intentions map cleanly to structure. Most themes make that mapping unnecessarily complicated.

The problems tend to cluster in three places: pattern organisation, template clarity, and typographic scale. Each one is a design decision that the theme makes on behalf of the editor — and most of the time, the decision is wrong.

Pattern organisation

Block patterns are the most powerful tool in a theme author's kit. They let you define reusable sections — heroes, feature grids, testimonials — that editors can insert without thinking about layout. But most themes ship patterns in a single undifferentiated list, with names like "Pattern 1" and "Pattern 2."

A well-organised pattern library has categories. It has clear names. It has patterns that do one thing well, not patterns that try to anticipate every possible use case. The editor should be able to look at the inserter and immediately understand what they're choosing.

Template clarity

WordPress block themes use templates to define the structure of different page types. A well-built theme has a small, coherent set of templates — perhaps six or eight — each with a clear purpose and a clear relationship to the others.

What editors actually encounter in most themes is a confusing array of templates with overlapping purposes, inconsistent padding, and no documentation. The editor opens the site editor, sees twenty templates, and has no idea which one controls the page they're trying to edit.

The typographic scale problem

Typography in the block editor is controlled through two mechanisms: the global styles in theme.json, and the per-block typography controls in the editor sidebar. Most themes set up the global styles correctly but leave the per-block controls unconstrained — which means editors can set headings to any size they want.

This is not editorial freedom. It is design entropy. The right approach is to define a fixed type scale in theme.json and expose only those sizes to the editor. Three or four sizes is enough. More than that and the system starts to break down.

What a well-built theme looks like

A well-built block theme has the following properties. The pattern library is organised into categories with clear names. Templates are documented and purposeful. The type scale is fixed and exposed in full. Spacing uses a consistent grid — in my case, an 8px base unit — applied everywhere without exception.

It also has something harder to define: a clear editorial point of view. The theme should make good layout decisions the default, not the exception. The editor should be able to insert a pattern, fill in the content, and trust that the result will be coherent.

/* theme.json — type scale definition */
"typography": {
  "fontSizes": [
    { "name": "Small", "slug": "sm", "size": "0.889rem" },
    { "name": "Base", "slug": "base", "size": "1rem" },
    { "name": "Large", "slug": "lg", "size": "1.25rem" },
    { "name": "XL", "slug": "xl", "size": "1.563rem" }
  ]
}

The code above is a simplified version of the type scale I use in every block theme. Four sizes. No freeform input. The editor sees exactly what the designer intended — nothing more, nothing less.

The deeper point

The reason most WordPress themes fail their editors is not technical. It is a failure of empathy. The theme author optimises for the moment of building, not the moment of editing. The features that make a theme impressive in a demo — elaborate animations, complex layout options, extensive customisation — are often the same features that make it hard to maintain.

A theme that constrains well is a theme that ages well. The editor who inherits a site two years after launch should find it as coherent and navigable as the day it went live. That is the goal. Everything else is decoration.
